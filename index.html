목표 소개

사용자들이 게시글 작성에서 다중사진을 업로드 할 수 있는 CRUD - BackEnd

1-1. Multipart를 List로 사용
○ 사용자들이 SFlash에서 여러장의 사진으로 해당 스팟에 대한 정보를 얻을 수 있도록 기능을 구현하였습니다.
○ 게시물 수정 기능에서도 사용자들이 업로드한 사진을 삭제 또는 다른 사진을 추가할 수 있도록 구현하였습니다.
■ 여기서 삭제는 DB에 저장된 이미지 URL만 삭제하는 것이 아니라 S3에서 제공하는 DeleteObjectsRequest 를 사용하여 S3에 업로드된 이미지도 삭제하여 S3의 용량도 확보하였다

사용자의 편의성 개선(소셜로그인, 페치조인, 무한 스크롤) - BackEnd

2-1. 소셜 로그인 API 서비스를 사용하였습니다.
○ 안전하고 신뢰할 수 있는 방법으로 사용자들이 가입할 수 있게 네이버, 카카오, 구글 소셜로그인 구현하였습니다.
○ 이미 이메일이 가입된 경우 소셜로그인을 사용시 같은 유저로 볼 수 있도록 구현하였습니다.
○ provider가 naver, google, kakao 마다 다른 userInfo를 만들어서 받을 수 있도록 구현하였습니다.(각 소셜로그인마다 보내주는 변수명을 다르게 하였습니다.)
2-2. /map(GetMapping), /board(GetMapping)에 대한 API를 호출할 때 시간 최적화 - BackEnd
○ /map, /board를 호출할 때 Board, User, Heart, BoardImgUrl에 대한 정보를 한번에 보내줘야한다. 그러면 한명의 User에 대한 여러개의 Board중에서 한 개의 Board에 대한 여러개의 이미지, 좋아요를 DB에서 조회하는 비용이 어마어마하다. 그래서 이 부분을 반복문을 사용하면서 db에 접속했지만 배치조인을 사용하여 한번에 조회할 수 있도록 코드를 수정하여 API호출 시간을 줄였습니다.
2-3. 무한 스크롤 기능 구현 - BackEnd
○ 커뮤니티와 마이페이지에 무한스크롤 방식을 적용하여 데이터가 방대해져도 페이지가 로딩되기를 기다리지 않고 정보를 탐색할 수 있도록 구현하였습니다.
○ 처음 값은 임의의 큰 수인 9999로 설정하였습니다.
○ 요청 파라미터로 lastId와 size를 받아서 lastId보다 작은, size만큼의 게시물들을 반환하였습니다.

무중단 CI/CD - BackEnd

3-1. 개발자 배포과정에서의 편의성 개선
○ 기존의 FileZila로 배포를 했을 때는 팀원들이 Push를 하게되면 Merge와 동시에 build, *.jar파일 옮기기, ec2에 옮겨서 배포하기 라는 시간 비용이 들었다.
○ Travis,스크립트,nginx 활용하여 무중단 배포를 구현하였고 이 결과 코드를 Push 하게 되면 배포와 서버의 작동이 자동화가 되기 때문에 개발자의 편의와 시간이 줄어들었습니다.

지도를 활용한 사용자 인터페이스 개선 - FrontEnd

1. 사용자의 현재접속위치를 이용하는 geolocation 기능을 사용.
○ 사용자가 자신의 주변 스팟을 쉽게 확인하고, 지도상에서 현재위치로 쉽게 이동할 수 있는 기능을 구현하기 위해서 html5에 추가된 자바스크립트 API인 geolocation을 이용
○ geolocation은 보안상의 이유로 보안 프로토콜을 지원하는 https에서만 가능하므로 https 도메인으로 배포함
○ https로 배포할 수 있도록 AWS에서 SSL인증서를 받고 그 인증서로 AWS의 클라우드프론트에 설정을 해주었음
